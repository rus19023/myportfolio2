Inmon Methodology

Bill Inmon says


            “one of the great appeals of the data warehousing concept is there is a single version of the truth”.

Mr. Inmon's ideas for data warehouses work on a "bottom-up" principle. His model starts with the enterprise's data model, specifically what is most important for the business to run. The structure of the data from varied sources will be integrated and normalized to avoid duplication of data, help identify and prevent data anomalies. Once the infrastructure is in place, the data will be collected, analyzed, and stored in data marts built for each business purpose needed, possibly denormalized to facilitate reporting for decision-making. The warehouse is the only source of data for the data marts.

The Inmon model is best for large-scale data warehouses. These data warehouses do take many months to design and build, so they are not suitable for an off-the-cuff solution. It will be an expensive, long-term investment, but worth it in the long run. It should be more secure, as there are no external or non-standardized data collections. There will be costs in the future upkeep, as the data will grow and more employees will be needed to support the data warehouse.

Kimball Methodology

Ralph Kimball had different thoughts about big data storage and analysis. He was concerned that if the data was normalized, it would be difficult to analyze. He begins with finding the purpose for each main business purpose, and builds the marts for each of these purposes. The data warehouse is a conglomerate of these data marts built around the central repository of raw data, which comes from varies sources. Each of these marts becomes a point in the "star" pattern of the data warehouse.

Kimball's methodology is more suitable for small-scale data warehouses. These data warehouses are designed to be done quickly, and are suitable for a quickly needed project. They will be much less expensive and completed much faster. There will be minimal costs for future upkeep.

OLTP-Online Transactional Processing

Online Transactional Processing (OLTP) is a term for the process of processing transactions in a database. Historically, database transactions have been considered those related to business, such as financial transactions at a bank, purchases at a store, removing shipped products from inventory, ticket booking systems, etc. A newer definition of transactions would include any process that changes the state of the database. The integrity consistensy and stability of the system, data and its relationships are the primary goals of OLTP, as the parties affected by any downtime are the end users, customers or clients.

This type of data structure is not well-suited to reporting or data analysis. Data optimization is geared toward improving the efficiency of the data processing and reporting processes. While the Inmon model is mostly used for this type of system, it can also work for OLAP systems that draw from OLTP data.

OLAP-Online Analytical Processing

Online Analytical Processing (OLAP) is very different than OLTP. It is more the use case for specific data marts. Data is generally normalized less than in OLTP, or not at all. Redundancy abounds, with multiple sets of the same data that are sorted by different criteria, used to create reports more efficiently than can happen in relational models.

There are also hybrid systems that combine these two with additional layers of applications. This type of system can support either Inmon's or Kimball's models.
